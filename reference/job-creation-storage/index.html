    Создание Job и где что лежит¶

    Простыми словами: как в OurPlanCore создаётся job, куда программа кладёт листы (sheets) и takeoff'ы, и почему они не теряются. Эта страница — для человека, которому надо понять «где мои файлы», а не для разработчика.

    Главная идея за 10 секунд

    Job — это одна папка на диске. Всё внутри неё: исходные PDF, листы, takeoff'ы, AI-контекст. Ничего в облаке. Скопировал/забэкапил эту папку — сохранил всю работу. Удалил её — потерял всё. Поэтому job-папку не трогают руками: не переименовывают файлы внутри, не двигают листы между папками проводником.

    Как создать job¶

    Всё начинается с кнопки Open / Import на ленте (или Ctrl+O). Четыре способа:

    Способ Что выбрать Что происходит
    Из папки с PDF New job from a folder of PDFs... Указываешь папку с PDF → программа ищет *.pdf (включая подпапки), просит имя job, создаёт его и сразу импортирует все PDF. Имена листов не спрашивает.
    Пустой job Blank job — start empty, add sheets later Создаёт пустой job без листов. Листы добавишь потом (Add Pages, Blank Sheet).
    Из PDF takeoffs Import PDF Takeoffs... Импортирует PDF вместе с замерами-аннотациями (lines/areas/counts) как редактируемые takeoff'ы. Сначала превью, файлы пишутся только после подтверждения.
    Из PlanSwift PlanSwift project... Конвертирует проект PlanSwift в job. Исходник PlanSwift открывается только на чтение.

    Добавить листы в уже открытый job

    Add Pages — импорт PDF · Blank Sheet — пустой лист · PDF Takeoffs — PDF + аннотации. Через меню Open / Import → Import into the current job есть PDF file(s)... (спрашивает имена листов), Folder of PDFs... (не спрашивает) и PlanSwift project....

    Импорт в существующий job¶

    Когда job уже открыт и нужно добавить ещё листы — заходишь в Open / Import → Import into the current job. Четыре пути, и каждый кладёт файлы в своё место внутри job:

    Что выбрать Спрашивает имена? Куда ложатся листы / takeoff'ы
    PDF file(s)... Да — даёшь имя каждому листу <job>\Pages\00. imported\<имя>\
    Folder of PDFs... Нет — берёт имена из файлов, ищет рекурсивно <job>\Pages\00. imported\<имя>\
    PlanSwift project... Нет <job>\Pages\01. planswift\ и <job>\Takeoffs\01. planswift\
    Import PDF Takeoffs... → Import into current job Нет листы → <папка импорта>\from pdf, takeoff'ы → <родитель takeoff>\from pdf

    Те же действия есть на ленте кнопками: Add Pages (= импорт PDF), Blank Sheet (пустой лист), PDF Takeoffs (PDF + аннотации).

    PDF — файлы или папка¶

    1. Open / Import → Import into the current job → PDF file(s)... (или Folder of PDFs...).
    2. PDF file(s)... — выбираешь один/несколько PDF, программа спрашивает имя каждого листа. Folder of PDFs... — указываешь папку, она ищет *.pdf рекурсивно и импортирует без вопросов про имена.
    3. Появляется диалог Import PDF Options с галочкой растра (см. ниже).
    4. Каждый PDF копируется в <job>\sources\, под Pages\00. imported\ создаются папки-листы, дерево Pages перечитывается.

    PlanSwift в текущий job¶

    1. Open / Import → Import into the current job → PlanSwift project....
    2. Указываешь проект PlanSwift — он открывается только на чтение, исходник не меняется.
    3. Страницы PlanSwift конвертируются из картинок в PDF-листы и ложатся в Pages\01. planswift; takeoff'ы — в Takeoffs\01. planswift.
    4. По умолчанию страницы PlanSwift без замеров пропускаются. Чтобы затащить всё — включи Import all PlanSwift sheets and takeoff folders.

    PlanSwift двумя путями

    Кнопка PlanSwift на ленте делает отдельный конвертированный job. Import into the current job → PlanSwift project... — добавляет в уже открытый job (в папки 01. planswift). Не путай.

    PDF Takeoffs в текущий job¶

    1. PDF Takeoffs на ленте (или Open / Import → Import PDF Takeoffs...).
    2. Выбираешь папку с PDF, где есть замеры-аннотации.
    3. Режим — Import into current job.
    4. Галочки опций: импортировать lines/areas/counts как редактируемые takeoff'ы; импортировать размеры как Ruler-аннотации; убрать аннотации из фона листа.
    5. Программа сканирует PDF, показывает превью и пишет файлы только после подтверждения. Листы → …\from pdf, takeoff'ы → …\from pdf, отчёт — в <job>\import_reports.

    Галочка «растр» при импорте PDF¶

    При любом импорте PDF (новый job из папки, Add Pages, PDF file(s)/Folder of PDFs) выскакивает окно Import PDF Options:

    Import N PDF page(s). ☐ Build readable raster cache and strict black-line snap index Original PDFs stay as the source for export, metadata, layers, and rebuild.

    Что делает галочка¶

    • Включена → во время импорта программа дополнительно «запекает» каждый лист в растровую картинку (~200 DPI) и строит snap-индекс по чёрным линиям. Кэш лежит рядом с листом: <job>\Pages\…\<лист>\raster\ (+ snap.json).
    • Выключена → импорт быстрее и без лишних файлов, но лист рисуется «налету» через PDF-рендер, и точная привязка к чёрным линиям недоступна.

    Зачем это нужно — простыми словами

    Растровый кэш делает тяжёлые PDF быстрее в открытии и включает строгий snap к чёрным линиям чертежа (аккуратнее обводить стены/контуры). Платишь за это временем импорта и местом на диске (картинки в папке raster\).

    Важно¶

    • Оригинал PDF остаётся источником. Растр — это только кэш поверх. Экспорт, метаданные, слои и пересборку программа по-прежнему делает из PDF в sources\. Удалишь папку raster\ — лист не сломается, просто потеряешь быстрый кэш.
    • Выбор запоминается. Галочка сохраняется в настройках (BuildRasterCacheOnPdfImport) и в следующий раз стоит так же, как оставил.
    • После импорта в итоговом сообщении видно Raster built: X, failed: Y.

    Путь — как пользоваться¶

    1. Open / Import → выбрать любой PDF-импорт (или Add Pages).
    2. В окне Import PDF Options:
    3. большие/тяжёлые чертежи, нужен точный snap по линиям → поставь галочку;
    4. быстрый черновой импорт, важна скорость → сними галочку.
    5. Import. При включённой галочке в прогрессе будет building raster k/n... — это нормально, идёт построение кэша.

    Куда сохраняется сам job¶

    Когда создаёшь job, программа решает, в какую корневую папку его положить, по порядку (берёт первое доступное):

    1. выбранный jobs root из Job Picker;
    2. родитель текущего job;
    3. сохранённый JobsRootPath / недавние roots из настроек;
    4. Documents\OurPlaneCore Jobs;
    5. запасной вариант — Desktop\OurPlaneCore Jobs.

    Где смотреть, если не помнишь

    Путь к последнему job и список недавних хранятся в %APPDATA%\OurPlaneCore\settings.json (поля LastJobPath, JobsRootPath, JobsRootPaths, RecentJobs). Самое простое — Open / Import → Recent (Ctrl+Shift+O).

    Что появляется на диске¶

    Один job — такая структура папок:

    <job>\
├─ Data.xml              ← карточка самого job (имя, тип, GUID)
├─ sources\              ← КОПИИ всех исходных PDF
│   └─ Arch.pdf
├─ Pages\                ← ЛИСТЫ (sheets)
│   └─ 00. imported\
│       └─ A-101\        ← один лист = одна папка
│           ├─ Data.xml        ← видимое имя листа и порядок
│           ├─ source.json     ← ★ главный файл листа
│           ├─ source_pdf.json ← необязательно (метаданные)
│           ├─ layers.json     ← необязательно (слои)
│           └─ annotations.json← необязательно (аннотации/линейки)
├─ Takeoffs\             ← TAKEOFF'ы
│   └─ Walls\
│       └─ Exterior 2x6\ ← один takeoff item = одна папка
│           ├─ Data.xml         ← свойства (цвет, тип, цена, notes)
│           └─ measurements.json← ★ вся геометрия замеров
├─ AI_Context\           ← AI-наблюдения, кропы, запросы
├─ import_reports\       ← отчёты импорта (необязательно)
└─ bookmarks.json        ← закладки видов (необязательно)

    Про имена папок

    Имя папки на диске ≠ видимое имя в программе. На диске имя «причёсано» и сделано уникальным, а человекочитаемое имя лежит в Data.xml. Поэтому не ориентируйся только на имя папки в проводнике.

    Где лежат листы (sheets)¶

    Три вещи, которые надо понять про листы:

    1. Исходные PDF копируются внутрь job. При импорте каждый PDF копируется в <job>\sources\<имя файла>.pdf. После импорта оригинал больше не нужен — программа работает с копией внутри job. Можно перенести job на другой компьютер, и листы откроются (оригинальная папка с PDF не требуется).

    2. Лист — это папка с source.json. Папка под Pages\ становится листом только потому, что в ней лежит source.json. Это главный файл листа: в нём относительный путь к PDF, номер страницы (с нуля), масштаб и флаги.

      {
  "pdf": "..\\..\\sources\\Arch.pdf",
  "page": 0,
  "scale_m_per_pt": 0.004233333333333333,
  "hidden_takeoffs": []
}

      Путь pdf — относительный от папки листа; при открытии программа сама разворачивает его в полный путь. Многостраничный PDF даёт несколько папок-листов, которые ссылаются на один и тот же скопированный PDF, отличаясь полем page.

    3. Сломать source.json = лист не откроется. А вот source_pdf.json — необязательный (метаданные для rename/scale); если его удалить, лист всё равно откроется.

    Не редактируй пути в source.json руками

    Путь pdf относителен папки листа. Если поправить его «на глаз», лист перестанет находить свой PDF.

    Где лежат takeoff'ы¶

    Takeoff'ы не лежат внутри листа. Они живут отдельно, под <job>\Takeoffs, и привязываются к листу ссылкой. Два типа узлов:

    • Папка (SmartNodeKind=folder) — группировка, например Walls.
    • Item (SmartNodeKind=item) — конкретный takeoff: цвет, тип замера, символ счёта, цена, notes. Вся геометрия замеров лежит в <job>\Takeoffs\<item>\measurements.json.

    measurements.json — массив замеров. У каждого замера есть ключевое поле page_folder — это и есть связь takeoff → лист: путь к папке листа, которому принадлежит замер.

    Как это работает при открытии job

    Программа грузит все takeoff items и все их замеры в память. Потом, когда открываешь конкретный лист, вьюпорт рисует только те замеры, у которых page_folder совпадает с открытым листом. Поэтому один takeoff item может иметь замеры на разных листах — каждый замер сам «знает» свой лист.

    Простыми словами: почему ничего не теряется¶

    • Листы не исчезают, потому что их PDF скопированы внутрь job (sources\), а каждый лист описан своим source.json. Пока папка job цела и source.json не тронут — листы открываются, даже если оригинальные PDF удалить.
    • Takeoff'ы не исчезают, потому что они лежат в Takeoffs\ своими папками с measurements.json, а поле page_folder помнит, на каком листе их показывать.
    • Сохраняй/бэкапь всю папку job целиком. Не выдёргивай отдельные файлы. Один job = одна самодостаточная папка.

    Если листы или takeoff'ы «пропали»¶

    Симптом Причина Что делать
    Takeoff'ы есть на диске, но не видны на листе Job скопировали в другое место → в measurements.json остались старые абсолютные page_folder Кнопка Repair Links (панель Pages) — переподключает замеры к папкам листов
    Лист не открывается Сломан/удалён source.json или копия PDF в sources\ Восстановить source.json / PDF; не править пути вручную
    Лист открылся, но «съехали» метаданные Удалён source_pdf.json Не критично — лист работает; метаданные можно пересобрать
    Не тот масштаб у замеров scale_m_per_pt в source.json / у замера Перезадать масштаб инструментом Scale

    Чего не делать руками в папке job

    • Не переименовывать и не двигать файлы внутри job проводником.
    • Не править пути в source.json и page_folder в measurements.json.
    • Не класть геометрию замеров в папку листа — программа ждёт её только в папке takeoff item.

    Быстрый чек-лист: проверить job на диске¶

    • В корне job есть Data.xml, Pages, Takeoffs, sources.
    • У выбранного листа в Pages\...\ есть source.json.
    • Путь pdf из source.json (относительно папки листа) ведёт к существующему PDF в sources\.
    • У takeoff item в Takeoffs\...\ есть measurements.json.
    • У каждого замера page_folder указывает на существующую папку листа.
    • Если takeoff не виден — сравни путь открытого листа, page_folder замера и видимость (hidden_takeoffs).
